# CookLikeHOC 厨师 Agent 需求分析文档

---

## 一、项目概述

### 1.1 项目背景

[CookLikeHOC](https://github.com/Gar-b-age/CookLikeHOC) 是一个开源仓库，收录了老乡鸡《菜品溯源报告》中公布的全部菜品配方。该仓库以 Markdown 文件形式，按菜品分类（主食、卤菜、炒菜、汤等）整理了 **200+** 道菜品的原料、配料、制作步骤和营养成分。

然而，当前厨师在使用时面临以下痛点：
- **查找效率低**：需要手动翻阅文件目录，无法快速定位目标菜品
- **无法模糊搜索**：只能按文件名查找，不能按食材反查菜品
- **缺少智能推荐**：没有"每日推荐""根据现有食材推荐菜品"等能力
- **配方难以对比**：同类菜品之间缺乏横向对比

### 1.2 项目目标

构建一个 **基于 LLM 的智能厨师 Agent**，支持：
1. **自然语言查询**：厨师用自然语言问"阳春面怎么做？"即可获得完整配方
2. **食材反查**：问"家里有鸡腿，能做什么菜？"自动匹配菜品
3. **每日推荐**：根据时令、营养搭配智能推荐菜品组合
4. **配方对比**：对比多个菜品的原料差异

### 1.3 项目范围

| 模块 | 说明 |
|------|------|
| 数据导入 | 从 CookLikeHOC 仓库拉取所有菜品 Markdown，解析入库 |
| ETL 处理 | 将 Markdown 清洗、结构化，存入关系数据库 + 向量数据库 |
| RAG 检索 | 基于向量数据库实现语义检索，支持食材反查 |
| LLM Agent | DeepSeek-V4-Flash 驱动，ReAct 模式，支持 Tool Call |
| API 服务 | RESTful API，供前端/小程序/企微机器人调用 |
| 用户系统 | 扩展功能 — 多厨师账号、收藏、使用历史 |

## 二、用户角色分析

### 2.1 角色定义

| 角色 | 需求场景 | 核心诉求 |
|------|---------|---------|
| **门店厨师** | 日常做菜时查配方、看用量 | 快速、准确、语音输入 |
| **中央厨房研发** | 对比配方、调整用料比例 | 精确数据、对比分析 |
| **门店店长** | 规划当日菜单、控制食材成本 | 菜品推荐、库存匹配 |
| **系统管理员** | 管理菜品数据、用户账号 | 数据导入、权限管理 |

### 2.2 用户故事

- 作为厨师，我希望能 **用自然语言问** "阳春面的汤底怎么调？"，Agent 直接告诉我配方和用量
- 作为厨师，我希望能 **说出我有的食材**，Agent 推荐我能做的菜
- 作为厨师，我希望能 **按分类浏览** 所有主食/卤菜/汤品
- 作为店长，我希望每天收到 **基于季节和营养的推荐菜单**
- 作为管理员，我希望能一键 **增量更新** CookLikeHOC 仓库的新菜品

## 三、功能需求

### 3.1 核心功能

#### F1 — 菜品智能问答（RAG）

| 属性 | 描述 |
|------|------|
| 描述 | 用户用自然语言提问，Agent 从知识库检索相关菜品，结合 LLM 生成回答 |
| 输入 | "老母鸡汤怎么炖？" / "卤鸡腿的卤料配方是什么？" |
| 输出 | 菜品名称、配料清单（含用量）、制作步骤、营养成分 |
| 技术 | Embedding → Milvus/Qdrant 向量检索 → LLM 生成 |

#### F2 — 食材反查

| 属性 | 描述 |
|------|------|
| 描述 | 用户输入手头食材，Agent 返回匹配的菜品列表 |
| 输入 | "鸡腿、花椒、辣椒" |
| 输出 | 卤鸡腿、口水鸡、辣子鸡...（按匹配度排序） |
| 技术 | 食材名称提取 → 向量匹配或全文检索 → LLM 排序 |

#### F3 — 每日智能推荐

| 属性 | 描述 |
|------|------|
| 描述 | 根据时令季节、营养均衡，每日推荐 3-5 道菜品的组合 |
| 输入 | 无（自动）或指定约束（"要清淡的"） |
| 输出 | 推荐菜单 + 推荐理由 |
| 技术 | 预设规则（季节映射）+ LLM 排序 |

#### F4 — 分类浏览

| 属性 | 描述 |
|------|------|
| 描述 | 按菜品分类（主食/卤菜/炒菜/汤等）浏览所有菜品 |
| 输入 | 分类名称或"全部" |
| 输出 | 该分类下所有菜品列表（含基本信息） |

#### F5 — 配方对比

| 属性 | 描述 |
|------|------|
| 描述 | 同时查询多个菜品的配方，进行横向对比 |
| 输入 | "对比卤鸡腿和口水鸡的用料差异" |
| 输出 | 并排对比表：相同原料 / 不同原料 / 用量差异 |

### 3.2 扩展功能

#### F6 — 用户系统

| 功能 | 描述 |
|------|------|
| 用户注册/登录 | JWT 认证，支持账号密码 + 手机号 |
| 角色权限 | 厨师 / 店长 / 管理员，RBAC 控制 |
| 收藏夹 | 收藏常用菜品，快速访问 |
| 查询历史 | 记录查询历史，支持回溯 |
| 自定义配方笔记 | 厨师可在菜品上添加个人备注（"本店减盐 2g"） |

#### F7 — 数据管理后台

| 功能 | 描述 |
|------|------|
| 菜品同步 | 从 CookLikeHOC GitHub 仓库增量拉取更新 |
| 菜品编辑 | 手动修改/新增菜品数据 |
| 数据校验 | 检查菜品字段完整性 |
| 知识库重建 | 触发向量数据库全量/增量重建索引 |

```
用户输入: "老母鸡汤怎么炖，要加什么料？"

    ▼
┌─────────────┐
│   Thought    │  → 分析意图：用户想查老母鸡汤的配方
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Action     │  → 调用 searchRecipe("老母鸡汤")
│  (Tool Call) │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Observation │  → 返回：[老母鸡汤, 肥西老母鸡汤, 金牌老母鸡汤...]
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Thought    │  → 用户问的是基础款"老鸡汤"，选第一个结果
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Action     │  → 调用 getRecipeDetail("老鸡汤")
│  (Tool Call) │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Observation │  → 配料：老母鸡、盐、鸡油
│              │    步骤：500g鸡 + 16g盐 + 60g鸡油 + 2500g水，炖60分钟
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Answer     │  → 老母鸡汤的配方是：每500g老母鸡，加16g盐、60g鸡油、
│              │    2500g水，烧开后炖制60分钟。出锅前用2g盐揉搓鸡表皮。
└─────────────┘
```

## 六、数据模型设计

### 6.1 菜品数据元模型

基于 CookLikeHOC 仓库中 Markdown 的现有结构，抽象出以下数据模型：

```java
// 菜品实体
public class Recipe {
    Long id;              // 主键
    String name;          // 菜品名称 (e.g. "阳春面")
    String category;      // 分类: 主食/卤菜/炒菜/汤/凉拌/早餐/炸品/烤类/炖菜/蒸菜/砂锅菜/烫菜/煮锅/饮品/配料
    String alias;         // 别名 (e.g. "老母鸡汤面" 也叫 "肥西老母鸡汤面")
    String imageUrl;      // 图片路径或URL
    String summary;       // 一句话简介
    String nutritionJson; // 营养成分 JSON (热量/蛋白质/脂肪/碳水/钠)
    List<Ingredient> ingredients;  // 配料列表
    List<Step> steps;              // 制作步骤列表
    String rawMarkdown;   // 原始 Markdown 内容 (用于 RAG 检索)
    LocalDateTime createdAt;
    LocalDateTime updatedAt;
}

// 配料实体
public class Ingredient {
    Long id;
    Long recipeId;        // 所属菜品ID
    String name;          // 原料名称
    String brand;         // 品牌/供应商 (e.g. 金寨先徽、长沙彩云)
    String quantity;      // 用量 (e.g. "15g", "500g")
    String note;          // 备注 (e.g. "依个人口味适量调整")
}

// 制作步骤
public class Step {
    Long id;
    Long recipeId;
    Integer order;        // 步骤序号
    String description;   // 步骤描述
}
```

## 七、ETL 处理流程

### 7.1 整体流程

```

        │
        ▼

        │

│ 2. Transform (转换)│

```

### 7.2 Markdown 解析规则

CookLikeHOC 的 Markdown 文件有 **两种** 主要格式：

**格式 A（标准化格式）**：
```markdown
# 菜品名称
![菜品名称](../images/菜品名称.png)

## 配料
- 原料A
- 原料B（供应商）

## 步骤
- 1. 步骤一描述；
- 2. 步骤二描述。

## 营养成分
| 项目 | 每 100g 含量 |
| :--- | :--- |
| 热量 | XX Kcal |
```

**格式 B（简化格式，如老鸡汤）**：
```markdown
# 老鸡汤
## 原料：
- 老母鸡
- 盐

## 步骤：
- 每只鸡...
```

### 7.3 向量化策略

| 参数 | 策略 |
|------|------|
| Embedding 模型 | `shibing624/text2vec-base-chinese` 或 `BAAI/bge-large-zh-v1.5` |
| 分块策略 | 按 **菜品** 为最小单元，每个菜品生成 1 个向量 |
| 向量内容 | 菜品名称 + 分类 + 配料列表 + 步骤摘要（拼接后 embedding） |
| 向量维度 | 768 (text2vec) / 1024 (bge) |
| 相似度算法 | Cosine Similarity |
| 重建策略 | 全量重建（ETL 全量同步时触发） |

### 7.4 SQL 建表语句（核心表）

```sql
CREATE TABLE recipe (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(128) NOT NULL COMMENT '菜品名称',
    category VARCHAR(32) NOT NULL COMMENT '分类',
    alias VARCHAR(256) COMMENT '别名',
    image_url VARCHAR(512) COMMENT '图片路径',
    summary VARCHAR(512) COMMENT '简介',
    nutrition_json JSON COMMENT '营养成分',
    raw_markdown MEDIUMTEXT COMMENT '原始Markdown',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_name_category (name, category)
) COMMENT '菜品表';

CREATE TABLE ingredient (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    recipe_id BIGINT NOT NULL COMMENT '菜品ID',
    name VARCHAR(128) NOT NULL COMMENT '原料名称',
    brand VARCHAR(256) COMMENT '品牌/供应商',
    quantity VARCHAR(64) COMMENT '用量',
    note VARCHAR(256) COMMENT '备注',
    FOREIGN KEY (recipe_id) REFERENCES recipe(id) ON DELETE CASCADE
) COMMENT '配料表';

CREATE TABLE recipe_step (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    recipe_id BIGINT NOT NULL COMMENT '菜品ID',
    step_order INT NOT NULL COMMENT '步骤序号',
    description TEXT NOT NULL COMMENT '步骤描述',
    FOREIGN KEY (recipe_id) REFERENCES recipe(id) ON DELETE CASCADE
) COMMENT '步骤表';
```

## 八、Agent 工具集设计

### 8.1 Tool 定义

Agent 通过 Tool Calling 机制调用以下工具，每个工具作为 Spring Bean 注册到 LangChain4j：

| 工具名 | 功能 | 输入参数 | 输出 |
|--------|------|---------|------|
| `searchRecipe` | 语义搜索菜品 | `query: String`, `topK: int = 5` | 菜品列表 + 相似度 |
| `getRecipeDetail` | 获取菜品详情 | `recipeId: Long` 或 `name: String` | 完整配方 |
| `reverseSearch` | 根据食材反查菜品 | `ingredients: List<String>` | 匹配的菜品列表 |
| `listCategory` | 列出某分类全部菜品 | `category: String` | 菜品列表 |
| `dailyRecommend` | 每日推荐 | `preference: String?` | 推荐菜品组合 |
| `compareRecipes` | 对比多个菜品 | `recipeIds: List<Long>` | 对比分析结果 |
| `getNutrition` | 查询营养成分 | `recipeId: Long` | 营养数据 |
| `webSearch` | 互联网搜索补充信息 | `query: String` | 搜索结果摘要 |

## 十二、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| CookLikeHOC Markdown 格式不统一 | ETL 解析失败率高 | 多格式兼容解析器 + 人工校验兜底 |
| DeepSeek API 不稳定 | 问答不可用 | 增加重试机制 + 备选 LLM（如 Qwen） |
| 向量检索精度不足 | 搜不到相关菜品 | 混合检索：向量 + ES 全文检索 + RRF 融合排序 |
| 菜品数据量增长 | 检索变慢 | Milvus 索引优化 + Redis 热点缓存 |
| LLM 幻觉 | 给出不存在或错误的配方 | Tool Call 约束 — 强制先从数据库检索再回答 |

## 十三、验收标准

| 编号 | 验收项 | 标准 |
|:---:|------|------|
| AC1 | 自然语言问答 | 问"阳春面怎么做"能返回正确配方（配料 + 步骤） |
| AC2 | 食材反查 | 输入"鸡腿、花椒"能返回卤鸡腿、口水鸡等 |
| AC3 | 模糊匹配 | 输入"鸡面"能找到"老母鸡汤面" |
| AC4 | 多轮对话 | 记住上下文：先问"有什么卤菜"，再问"第一个怎么做" |
| AC5 | 每日推荐 | 调用推荐接口返回 3-5 道菜品 + 推荐理由 |
| AC6 | ETL 完整性 | 同步后菜品覆盖率 ≥ 95%（排除 README 等非菜品文件） |
| AC7 | 响应时间 | 问答 P95 < 3 秒 |
